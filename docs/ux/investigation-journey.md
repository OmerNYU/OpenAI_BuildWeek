# Investigation journey

## Persona and success measure

- **Who:** a developer investigating a local React or Next.js failure.
- **Goal:** turn a report into an understandable, evidence-backed next step.
- **Success:** submit without guessing at optional fields; always know the current stage; distinguish analysis, execution evidence, and the verdict.

## Stages

### 1. Intake

The developer enters the repository and report. They may worry that the tool will change source code or needs a perfect report. The intake explains the trusted, clean-repository requirement and keeps technical extras optional.

### 2. Investigation

The developer waits for a response and needs confidence that work is progressing. The UI shows a current stage, elapsed time, an accessible progress indicator, and the ordered server timeline.

### 3. Review

The developer needs to know what was observed rather than infer success from a test exit code. Results label file-backed analysis, sanitized execution evidence, and the classifier verdict separately.

### 4. Recovery

If submission or polling fails, the error is announced, the last safe state remains visible, and the developer can start another investigation without a page reload.
